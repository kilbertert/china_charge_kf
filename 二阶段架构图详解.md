# 《二阶段架构图》详解文档

> **文档用途**：本文件对图片 `二阶段架构图.png` 中的所有元素（节点、形状、文字、连接线、分支条件、数据流向）进行逐项可重建级描述，使一个仅支持纯文本输入的模型能够完整还原该流程图的结构与业务语义。
>
> **图类型**：业务流程图（Flowchart）/ 状态机图
>
> **涉及语言**：简体中文
>
> **图形元素分类**：
> - **矩形**（实线边框）：处理节点（Process）
> - **菱形**（实线边框）：判断节点（Decision）
> - **矩形**（虚线边框）：特殊状态 / 缓存节点（Cached / Sub-state）
> - **带方向箭头线**：数据 / 控制流向
> - **箭头上的文字标签**：分支条件或事件说明

## 一、整体定位

这是一张**客户反馈处理与 Bug 跟踪系统**的二阶段对话流程图。

- **第一阶段**：客户反馈问题的接收 → 现有 Bug 数据匹配 → 反馈内容的整理与语言转写。
- **第二阶段**：基于反馈的修改处理 → 客户多轮确认 → 数据写入 / 修改 → 超时缓存与新对话识别。

整张图围绕"客户反馈是否已在 Bug 反馈数据表中存在"这一关键判断展开，并设计了三种典型分支：**新增反馈**、**已有行修改**、**无关新对话**。

## 三、连接线与分支条件（逐条说明）

> 标记约定：`A → B [标签]` 表示"从节点 A 流向节点 B，箭头上的标签为分支条件或事件名"。

### 3.1 起点：客户反馈问题（入口）

- **N1 客户反馈问题 → N2 查询是否在 bug 反馈数据表**
  - 无分支标签，作为入口直接进入查表判断。

### 3.2 第一阶段主分支：是否已存在记录

- **N2 查询是否在 bug 反馈数据表** 是一个菱形判断节点，共有两条带标签的出口：
  - **N2 → N5 整理反馈内容 转写内容语言**（标签：**N**）—— 表中不存在该问题，进入"新增反馈整理"路径。
  - **N2 → N3 整理并输出该行问题内容和进度**（标签：**相关性识别**）—— 表中存在相关记录，进入"已有记录处理"路径。

### 3.3 已命中记录路径（命中分支）

- **N3 整理并输出该行问题内容和进度 → N4 客户确认是否为该问题**
- **N4 客户确认是否为该问题** 菱形两条出口：
  - **N4 → N5 整理反馈内容 转写内容语言**（标签：**否认**）—— 客户否认是同一个问题，落回到"新增 / 重整理"路径。
  - **N4 → N9 旧行问题和客户所提内容对比**（标签：**确认 并可能携带：补充修改**）—— 客户确认是同一行问题，并且可能在确认时携带补充修改意见。

### 3.4 内容对比与"是否要修改"判定

- **N9 旧行问题和客户所提内容对比** 菱形三条出口：
  - **N9 → N10 结束对话**（标签：**静默**）—— 客户沉默（不表态）。
  - **N9 → N10 结束对话**（标签：**无差异**）—— 客户已提内容与旧行一致，无需修改，直接结束。
  - **N9 → N11 询问是否要对改行数据修改**（标签：**有差异**）—— 存在差异，进一步询问客户是否要改这一行。

- **N11 询问是否要对改行数据修改** 菱形三条出口：
  - **N11 → N12 完成修改并汇报内容**（标签：**确认修改**）—— 客户同意修改。
  - **N11 → N15 缓存对话内容并标记用户 / 并设时间留：半小时**（标签：**对话超时**）—— 客户未在限定时间内答复，超时缓存。
  - （图中 N11 还有一条向下与"修改该行 …"路径相连的备用连线，详见 3.7）

### 3.5 第二阶段：执行修改 & 客户二次确认

- **N12 完成修改并汇报内容 → N13 客户是否确认结果**（下方的"客户是否确认结果"）
- **N13 客户是否确认结果**（下方）菱形三条出口：
  - **N13 → N14 修改该行，并再次发送对话提醒表链接和编号，内容**（标签：**确认**）—— 客户确认修改无误，写入并再次推送。
  - **N13 → N15 缓存对话内容并标记用户 / 并设时间留：半小时**（标签：**对话超时**）—— 客户超时未确认，进入缓存窗口。
  - **N13 → N12 完成修改并汇报内容**（标签：**不确认 / 携带修改要求**）—— 客户不确认并提出新的修改要求，**回流到 N12 重新修改**（闭环反馈）。

### 3.6 缓存与"新对话识别"机制

- **N14 修改该行，并再次发送对话提醒表链接和编号，内容 → N15 缓存对话内容并标记用户 / 并设时间留：半小时**
  - 即修改动作本身完成后，仍要把这一轮对话挂到该用户的缓存中，并设置 **半小时（30 分钟）** 的存留窗口，便于后续同用户新对话进来时做相关性识别。

- **N15 缓存对话内容并标记用户 / 并设时间留：半小时 → N17 标记用户新对话：客户回应需要修改，相关性识别缓存存聊天**
  - 进入"用户级缓存标记"特殊状态。

- **N17 标记用户新对话 …（虚线框）** 自身是一个**分发器**，共有三个出口：
  1. **N17 → N5 整理反馈内容 转写内容语言**（标签：**无相关性识别**）
     - 当该用户在 30 分钟缓存窗内的新对话与原问题**不相关**时，跳过相关性识别，直接进入"新问题整理"路径。
  2. **N17 → N8 客户回应需要修改**（标签：**相关性识别 1 是新增数据修改**）
     - 当新对话被识别为"针对新数据的修改要求"，回流到"客户回应需要修改"分支。
  3. **N17 → N9 旧行问题和客户所提内容对比**（标签：**相关性识别 2 是旧行数据修改**）
     - 当新对话被识别为"针对已有旧行的数据修改"，直接进入"旧行内容对比"分支。

### 3.7 超时写入与"另起表缓存"

- **N11 询问是否要对改行数据修改 / N13 客户是否确认结果（下方）** 的"对话超时"分支最终都会汇集到 **N15 缓存对话内容并标记用户**，由缓存窗口决定下一步走向。
- **N15 → N18 超时写入**（来自上半段 N7 的"对话超时"出口）—— 第一阶段反馈结果确认时客户超时，也会进入超时写入分支。
- **N18 超时写入** 自身有一个分支：
  - 在图的下半部分，**N18 → N19 超时仍未更改完成旧行数据另起表缓存，不改数据表数据**
  - 含义：当旧行修改在缓存窗口内**仍未被客户确认完成**时，本次不写入主数据表，而是把未完成内容放到**另一张缓存表**中暂存。

### 3.8 第一阶段反馈结果确认（新增反馈路径）

- **N5 整理反馈内容 转写内容语言 → N6 整理回答记录反馈内容 / 引导客户回应反馈结果**
- **N6 → N7 客户是否确认结果**（上方的"客户是否确认结果"）
- **N7 客户是否确认结果**（上方）菱形三条出口：
  - **N7 → N16 写入数据表，并再次发送对话提醒表链接和写入编号，内容**（默认正向路径）—— 客户认可后正式写入新行。
  - **N7 → N6 整理回答记录反馈内容 / 引导客户回应反馈结果**（标签：**客户继续反馈对话**）—— 客户继续提出反馈，回流到 N6 形成多轮对话闭环。
  - **N7 → N15 缓存对话内容并标记用户 / 并设时间留：半小时**（标签：**对话超时**）—— 客户超时未确认，进入缓存。
- **N6 自身也可由 N8 客户回应需要修改 进入**（标签：**携带修改要求**）—— 即客户回复"需要修改"后，整理他的修改要求并把流程重新带回 N6，由 N6 引导客户对结果做出回应。

## 四、特殊形状与符号说明

1. **起点节点 N1 "客户反馈问题"**：作为流程入口，使用普通矩形表达，无前置输入。
2. **终点节点 N10 "结束对话"**：仅作为"无差异"或"静默"两种情形下的终点，不存在流出箭头。
3. **虚线矩形 N17**：表示这是**用户级缓存状态**，不是普通流程节点；它的作用是"标记 + 分发"，识别用户新进对话的相关性后再分发到三条不同路径。
4. **虚线矩形 N19**：表示**另一张表 / 二级缓存**，专门承接"超时且未完成修改"的旧行会话，**不污染主数据表**。
5. **半小时（30 分钟）时间窗**：N15 设置的时间留是**全局缓存有效期**，所有进入缓存的对话与标记都共享这个 30 分钟的窗口；窗口期内该用户的任何新对话都会被 N17 拦截并做相关性识别。

## 六、业务逻辑要点（便于文本模型抓住重点）

1. **三路分发入口**：
   - 入口 1（直接新增）：`客户反馈问题 → 查询数据表 → N(不存在) → 整理反馈内容 → ...`
   - 入口 2（命中已有行）：`客户反馈问题 → 查询数据表 → 相关性识别 → 整理输出 → 客户确认 → 对比 → 决定修改与否`
   - 入口 3（缓存用户的再激活）：`缓存 → N17 标记用户新对话 → 相关性识别 1/2 → 重新进入主流程`

2. **三处形成闭环**（回流箭头）：
   - N7 → N6（客户继续反馈对话）
   - N13 → N12（客户不确认，携带修改要求 → 重新完成修改）
   - N4 → N5（客户否认是同一问题 → 重新整理）

3. **超时机制统一汇集点**：所有"对话超时"分支最终都进入 **N15 缓存对话内容并标记用户**，再被 **N17 标记用户新对话** 接住，等待 30 分钟内该用户的下一次发言。

4. **旧行修改的"隔离写入"**：当旧行修改超时未完成时（**N19**），数据**不写回主数据表**，而是放在另一张缓存表，避免半成品污染主表。

5. **写入动作有两类**：
   - **新增行写入**：N7 → **N16 写入数据表**（写入编号）
   - **已有行修改**：N13 / N11 → **N14 修改该行**（不产生新编号）

## 八、可供文本模型直接复述的一句话总结

> 这是一张"客户反馈 → Bug 数据表查询 → 相关性识别 → 三路分发（新增 / 命中旧行 / 缓存再激活）→ 多轮确认 → 修改或写入 → 超时缓存 + 30 分钟相关性识别 → 必要时另起缓存表暂存"的客户反馈与 Bug 跟踪二阶段对话流程图。整张图的核心枢纽是 **缓存对话内容并标记用户**（N15）与 **标记用户新对话**（N17），所有超时分支都汇集到 N15，并由 N17 在 30 分钟内对该用户的下一次发言做相关性识别后，分发回主流程的三个不同入口。
