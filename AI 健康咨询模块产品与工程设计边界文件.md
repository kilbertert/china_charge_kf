# AI健康咨询模块产品与工程设计边界文件（精简版）

## 一、模块定位与核心原则
- **工具定位**：本模块是销售小程序中的AI健康初筛与分诊建议工具，而非在线问诊或医疗诊断工具。
- **AI的角色**：AI在系统中扮演“规则执行者”与“语言表达器”，严禁其自由发挥或随机生成健康结论。系统必须严格基于“固定知识库 + 固定量表 + 固定解决方案模板”的模型运行。
- **统一禁止事项**：所有场景下，AI绝不输出疾病确诊、药物处方、停药/换药/加药建议、治疗方案及产品疗效承诺。严禁宣称产品能治疗或根治疾病，或引导用户替代药物、不用就医。

---

## 二、三阶段递进开发策略（由简到繁）
系统采取“重点领域深做、非重点领域浅答”的原则，分三个阶段逐步推进，以控制合规风险并确保快速上线：

### 第一阶段：体检报告分析（当前核心）
- **核心逻辑**：基于客户上传的明确体检指标进行异常识别、风险排序与科普解释。由于有明确的报告指标作为依据，合规风险最低，最适合作为首发信任入口。
- **优先深做的优势赛道**：
  - 基础营养类：骨营养（骨密度、维D等）、肌肉营养、肠道营养（膳食纤维+益生菌）、皮肤营养
  - 问题解决类（代谢/视力/血糖/肝脏）：植物甾醇膳食胆固醇管理、越橘视疲劳用眼营养、益生菌血糖管理辅助、水飞蓟岩藻黄素肝脏健康支持
- **输出标准化结构**：一句话结论（判定严重程度）→ 可视化等级图（让客户一秒看懂位置）→ 后果教育（讲明为什么要重视）→ 引入成因筛查量表 → 提供基础健康与营养方向建议（非处方）→ 提示就医与复查

### 第二阶段：健康产品甄别（避坑助手）
- **核心逻辑**：针对客户上传的产品图片、原料名称或特定健康提问，帮助客户建立产品真假好坏的判断能力，主打“避坑”和建立专业信任，而非直接卖货。
- **知识库赋能**：清晰界定各类产品的核心成分、有效剂量范围、国际知名品牌原料商（如MenaQ7, Lonza UC-II等）以及常见夸大宣传的风险点。

### 第三阶段：健康问题初筛咨询（症状分诊）
- **核心逻辑**：针对客户直接描述身体不适（如“我腿疼”）的模糊输入，不做医疗诊断，只承担“安全漏斗”作用。
- **硬性原则**：此场景下因为无报告和医嘱，AI绝对不给处置意见、不给营养处方、不推荐产品。
- **输出内容**：仅包含风险甄别（危险信号筛查）、可能涉及的健康方向提示、建议就医科室及就医前的信息准备提醒。

## 三、核心场景设计与后台逻辑

### 场景一：体检报告分析（以骨密度为例）
- **判读与分级**：系统提取最低T值，自动映射分级：正常（≥ −1.0）、骨量减少（−1.0 ～ −2.5）、骨质疏松风险（≤ −2.5）。
- **原因分析量表（后台固定）**：引导用户回答是否绝经、是否有轻微摔倒后骨折史、日晒时长、膳食习惯等固定问题，严禁AI随机生成成因。
- **输出逻辑**：先给出判定问题程度的一句话结论，再用简单分级图展示所处水平，随后开展后果教育（强调骨折危害与重视理由），最后引导填写筛查表并根据得分调用固定的标准解决方案库。

### 场景二：身体不适初步甄别（以“我腿疼”为例）
系统不做“症状回答器”，而是“症状甄别器”，必须严格执行以下三层量表逻辑：
1.  **第一层：危险信号筛查（红色/医学优先级）**：排查突发剧烈疼痛、外伤后不能负重、一侧小腿明显肿胀发热、伴随胸闷心慌等紧急信号。一旦命中，AI触发紧急风险提示，不给任何营养建议，直接强制指导急诊就医并明确指出当前不要进行按摩等误操作。
2.  **第二层：疼痛定位与性质表**：针对无危险信号者，收集疼痛主要部位（膝盖、大腿等）、持续时间及加重诱因。
3.  **第三层：伴随症状与既往病史表**：关联尿酸、骨密度、慢病及用药史。
- **输出归类**：最终将症状归类到特定方向（如膝关节负重退变型、肌肉劳损型等），输出可能方向与就医准备信息，无任何调理或处置意见。

### 场景三：健康产品甄别与避坑
- **产品图片解析**：通过OCR识别核心成分与剂量，对比知识库标准，指出配方完整度（如钙片是否含维D）与标签夸大宣传风险。
- **原料知识科普**：明确说明常见有效剂量与国际认可的标杆原料供应商，帮助客户建立理性的健康管理观念。

## 四、非核心健康问题的“浅答”兜底规则
为了避免大面积出现“AI不会回答”的窘况，对于非核心、高风险领域（如肿瘤标志物、心电图异常、影像结节等），系统设置统一的通用浅答模板：
1.  明确告知该指标涉及较专业的医学判断，仅凭当前信息无法准确分析。
2.  给出指标的基础字面解释，但严禁展开深度诱因分析。
3.  清晰指明建议挂号咨询的对应医学专科。
4.  郑重声明本系统当前不提供疾病判断、治疗建议、营养方案或产品推荐。

## 五、工程师核心工程实现清单与技术架构
工程师在开发时应将其理解为：**意图识别 + 固定量表系统 + 结构化知识库调用 + 报告页面生成 + 大模型润色的确定性系统**。

### 后台必须建立的5类结构化知识库（严禁大模型盲目外挂）

知识库存放于 `kb_health_consult/` 目录，以 YAML 文件组织。Python 代码通过通用 YAML loader 读取，不硬编码任何业务数据。

| 编号 | 知识库 | 文件路径 | 内容 |
|------|--------|---------|------|
| 1 | 指标解释库 | `01_metric_explainer/*.yaml` | 指标名称、别名、分级阈值、人类可读结论 |
| 2 | 成因量表库 | `02_cause_questionnaire/questions.yaml` | 固定筛查问题、选项、code_key |
| 3 | 评分规则库 | `02_cause_questionnaire/scoring_rules.yaml` | 固定答案打分、语义推断规则、tag→solution 映射、优先级 |
| 4 | 标准建议库 | `03_standard_recommendation/recommendations.yaml` | 各风险等级的生活方式/营养/就医建议 |
| 5 | 风险等级库 | `04_risk_levels/grading.yaml` | T值/维生素D/血钙分级标准、urgent 信号与响应 |
| 6 | 禁止表达库 | `05_prohibited_expression/patterns.yaml` | P0/P1 禁止词正则、安全话术模板、豁免词 |
| 7 | 产品原料库 | `06_product_ingredient/ingredients.yaml` | 核心原料、有效剂量、标杆供应商、避坑项 |

1.  **指标解释库**：包含指标名称、异常分级规则、一句话结论模板、建议就医科室及禁止表达。
2.  **成因量表库**：包含各品类的固定筛查问题、选项映射的风险标签及权重。
3.  **标准建议库**：对应各风险等级与用户画像的固定生活方式和就医提醒文本。
4.  **产品原料库**：包含核心原料的作用方向、国际公认标杆原料商、成人有效剂量范围及避坑要点。
5.  **禁止表达库**：强力过滤词（如确诊、治愈、根治、保证有效、不用去医院等），做最终的安全审查。

### 技术流水线设计
用户自由输入 → 意图识别（场景分流） → 匹配深度模块/通用浅答 → 触发固定问卷交互（量表打标签） → 匹配风控分层（低/中/高/紧急风险） → 调取固定标准文本 → 大模型进行人话润色 → 前端一屏式报告页面渲染（柱图/饼图/优先排序） → 预留“转人工专业人员”入口。

## 六、内容生产与审核闭环流水线（总编审机制）
为解决项目推进中内容产出量大与个人医学细节精力有限的矛盾，采用AI流水线协作与单类别专家审核制：
1.  **素材海选（DeepSeek）**：利用DeepSeek针对各个细分健康问题广泛搜集多版本素材（医学版、科普版、量表版等）。
2.  **边界裁剪（ChatGPT）**：由ChatGPT严格按照小程序的商业边界、筛选标准（10条原则）进行过滤、重构，剔除诊断与疗效承诺。
3.  **科普复核（Gemini）**：由Gemini站在专家科普审稿角度，复核内容的安全性、通俗性、错漏及量表合理性。
4.  **专家签字入库（人工把关）**：第一阶段仅需聘请一位对应的专业医生（如骨科/内分泌科方向）和一位临床营养师，依照《专家审核表》对量表合理性、风险分级、表达合规性进行签批，通过后赋予版本号正式录入系统知识库。

> **文档信息**
> - 受众：产品经理、AI 工程师、合规顾问、前端开发
> - 更新频率：每阶段里程碑后 review 一次；场景规则变更时实时同步
> - 版本：v0.2 — 2026-06-27

## 附录 B：禁止表达库实现方案

### 目标

在所有场景输出进入前端之前，做最后一层合规过滤，确保不输出任何违反第六节禁止事项的表述。

### 触发位置

```
variable-aggregator (4090) → [禁止词过滤] → End (4099)
```

建议在 4090 和 4099 之间插入一个 code 节点 `safety_filter`，对 4090 的 `output` 字段做过滤后写入最终 output。

### 分级过滤规则

#### P0 — 命中即替换（整段输出替换为安全模板）

| 级别 | 关键词/正则 | 替换模板 |
|------|-----------|---------|
| P0 | 确诊、治愈、根治、保证有效、替代药物、不用就医、停药、换药、加药 | "本系统仅提供健康科普与初筛参考，不提供疾病诊断、治疗方案或药物建议。具体诊疗请遵医嘱。" |
| P0 | 有效率\d+%、治愈率\d+% | 同上 |
| P0 | 可以治愈、能够根治、保证治愈 | 同上 |

#### P1 — 命中打回重写（标记需 LLM 重新润色）

| 级别 | 关键词/正则 | 处理方式 |
|------|-----------|---------|
| P1 | 治疗、疗效、副作用小、推荐使用、建议服用、处方 | 在 output 中附加 `{"safetyFlag": "needs_rewrite", "hits": [...]}`，前端不展示原始文本，回传给 LLM 重写 |
| P1 | \d+(?:毫克|g|克|mg) 等具体剂量（未关联知识库标准剂量时） | 同上 |

### 实现代码（code 节点）

```python
import json
import re

P0_PATTERNS = [
    r"确诊", r"治愈", r"根治", r"保证有效", r"替代药物", r"不用就医",
    r"停药", r"换药", r"加药", r"有效率[达为]?\s*\d+%", r"治愈率[达为]?\s*\d+%",
    r"可以治愈", r"能够根治", r"保证治愈",
]
P1_PATTERNS = [
    r"治疗", r"疗效", r"副作用小", r"推荐使用", r"建议服用", r"处方",
    r"\d+\s*(?:毫克|g|克|mg)(?!\s*(?:以下|以内|以上|左右))",
]

SAFE_TEMPLATE = (
    "本系统仅提供健康科普与初筛参考，不提供疾病诊断、治疗方案或药物建议。"
    "具体诊疗请遵医嘱。"
)

def _match_any(text, patterns):
    hits = []
    for p in patterns:
        if re.search(p, text):
            hits.append(p)
    return hits

def main(output: str) -> dict:
    text = output or ""
    p0_hits = _match_any(text, P0_PATTERNS)
    if p0_hits:
        return {
            "output": SAFE_TEMPLATE,
            "safetyFlag": "replaced",
            "p0Hits": p0_hits,
        }
    p1_hits = _match_any(text, P1_PATTERNS)
    if p1_hits:
        return {
            "output": text,
            "safetyFlag": "needs_rewrite",
            "p1Hits": p1_hits,
        }
    return {"output": text, "safetyFlag": "clean"}
```

### 前端配合

- `safetyFlag = "replaced"` → 前端只展示替换后的安全模板，隐藏原始文本
- `safetyFlag = "needs_rewrite"` → 前端展示"正在优化表述..."，触发一次无用户输入的 LLM 重写请求
- `safetyFlag = "clean"` → 正常展示

### 命中日志

每次命中 P0/P1 时，记录以下字段用于后续优化 prompt：
- `session_id`、`scene`、`p0_hits` / `p1_hits`、输出前 200 字符快照
