## 二、架构图深度解读（我的理解）

### 2.1 这是一张什么图？

这是一张**面向 B 端 SaaS（充电桩运营平台）客服场景的 AI Agent 对话路由与处理架构**。它把"客户提出问题 → AI 判定 → 调取知识/数据 → 给出答复或转人工"的全过程**模块化、判定化、流程化**。

可以从三个视角理解它：

1. **业务流程视角**：一张端到端的客服对话 SOP（标准作业流程）。
2. **软件架构视角**：一张 AI Agent 的**决策与执行图**——意图识别是"路由器"，4 条路径是"处理器"，外挂记忆是"知识库/数据库"。
3. **数据流视角**：右侧的"钉钉/企微共享表格"是**唯一可信数据源**（single source of truth），主流程与各路径都从它取数。

### 2.2 整体结构（自顶向下 5 层）

| 层级 | 名称 | 关键节点 | 作用 |
|---|---|---|---|
| L0 | 入口 | 第一轮对话确认 | 触发整张图 |
| L1 | 主干路由 | 匹配功能 → 流程1 → 流程2 | **先把问题归类到"功能板块"** |
| L1.5 | 数据资产 | 外挂记忆 + 两张共享表 | 提供功能清单、bug 状态 |
| L2 | FAQ 快速通道 | 查询常见问题 | 高频问题秒答，不进意图识别 |
| L3 | 意图识别中枢 | 模块1：意图识别 | **4 路分发** |
| L4 | 4 个处理路径 | A/B/C/D | 真正的"办事"逻辑 |

### 2.3 关键设计模式

#### ① "先归类、再分意图"的两级路由
- **L1 的"流程1+流程2"**：先把问题归到"某个功能板块"（是否清楚 → 是否为基础功能）。
- **L3 的"模块1：意图识别"**：在功能板块内识别具体意图（4 类之一）。
- **为什么不一步到位？** 因为充电桩系统功能板块多（计费、订单、设备、场地、用户、财务...），**先缩范围到板块、再在板块内识别意图**，能极大降低 LLM 误判率，也便于用"功能清单表"做事实校验。

#### ② "基础功能 vs 增值功能"的二分流（流程2）
- **Y 分支（基础功能）**：进入 FAQ + 意图识别主路径。
- **N 分支（非基础功能）**：提示"需和产品和商务进一步沟通"，**说明这是项目级/客户级定制功能**。这是非常典型的 SaaS 客服处理逻辑：标准产品问题自助答，定制化问题转人工或商务线。

#### ③ "知识外挂 + 共享表格"作为记忆
- 用**钉钉/企微共享表格**而不是向量库/RAG，是因为：
  - 数据量小（功能清单、bug 列表，几十~几百条）。
  - 需要**人可编辑、可审计、可协作**（产品经理、客服主管都能改）。
  - 实时性高（bug 状态天天变，向量库重建太重）。
- 这是**轻量 RAG** 或**结构化检索**的实用化变体。

#### ④ 4 条意图路径的"业务对称性"

| 路径 | 用户痛点 | 数据源 | 产物形态 | 是否转人工 |
|---|---|---|---|---|
| **A 功能查找** | 找不到入口 | 无（菜单树） | 菜单路径列表（≤5 条） | 否，纯自助 |
| **B 业务规则** | 不懂逻辑 | 项目功能表格 | 业务说明 + 沟通建议 | **可能转产品** |
| **C 流程卡点** | 操作被拦截 | 流程定义（未在图中标数据源） | 缺失流程 + 沟通建议 | **可能转产品** |
| **D bug 追踪** | 系统缺陷/进度 | bug问题追踪表 | 录入或进度汇报 | **可能转研发** |

- B、C、D 三条路径的输出里都带"**是否需要产品沟通**"——意味着 AI 给出初步答案后，**必须主动询问用户要不要升级到人**。这是一个**人机协同**的兜底设计。

#### ⑤ 路径 D 的"查询-录入"二态
- D 路径有一个独特的**菱形判断**（"是否已存在该问题"）：
  - N（未记录）→ **写库**（记录新 bug）。
  - Y（已记录）→ **读库**（汇报项目进度）。
- 这是**"工单系统"模式的对话化复刻**：AI 既是用户入口，也是工单写入与查询的统一界面。

### 2.4 视觉与信息密度的隐含信号

- **绿色区 = 业务流**（所有对话/判断/输出）。
- **蓝色区 = 数据资产**（唯一一个独立的数据子系统，与流程区物理隔离）。
- **节点形状语义**：
  - 矩形 = 流程/输出动作。
  - 虚线矩形 = "对话输出/追问动作"（区别于确定性的"流程节点"）。
  - 菱形 = 判断。
- **模块 1（意图识别）单独占据中央位置且加粗**：它是整张图的**枢纽**，所有 4 个路径都从它出，是设计上的"分发器"。

### 2.5 与现有工作流对齐的优化方向（基于架构推断）

> 以下是基于架构图做出的合理推断，落地时需结合实际项目（AutoGenesis）现状调整。

1. **建立"功能板块字典"**
   - 把流程 1 + 流程 2 的判定结果落到一张**结构化功能板块表**（板块名/层级/是否基础功能/示例问题），由产品维护。
   - 当前 `docs/` 下的 `01-05` 文档是按子系统写的，**缺少一张"用户视角的功能板块字典"**，建议新建。

2. **实现"双级意图分类器"**
   - L1 = 板块分类（粗粒度，~10 类）。
   - L2 = 4 类细粒度意图（A/B/C/D）。
   - 两级可解耦：L1 用规则/小模型即可，L2 用 LLM 也无所谓，因为输入空间已被 L1 收窄。

3. **外挂记忆落地为"配置化检索"**
   - 用 **Excel/飞书表格** 作为权威源，AI 在调用时**只读**。
   - 给路径 B/D 各做一个**结构化查询器**（按板块/项目/状态过滤），避免 LLM 自由发挥。

4. **路径 A 的"菜单路径召回"实现要点**
   - 流程1 = 用 LLM 把用户问题拆成"短词条"（关键词抽取 + 同义词扩展）。
   - 流程2 = 在菜单树上做**多关键词路径检索**，结果**合并去重**，输出**前 5 条**。
   - 这正是图中"上限五个菜单以内"的硬约束。

5. **路径 D 的工单化**
   - N 分支写入 → 自动@对应研发/测试。
   - Y 分支读取 → 输出当前状态（待处理/处理中/已修复/已关闭）。
   - 建议和现有 bug 追踪系统（飞书/钉钉/Jira）打通。

6. **增加"转人工"出口**
   - 路径 B/C 的"是否需要产品沟通"、路径 D 的所有结果，都应该有**统一的"转人工/转工单"按钮**（在 Webview/IM 端实现）。

7. **首轮确认 → 后续多轮的记忆**
   - 图中"第一轮对话确认"是单向入口，**没有画多轮对话的回环**。实际落地需在 L1 之上加一层**会话状态机**，把"匹配功能"的过程做成多轮、可持续追问，而不是一次判定即结束。
