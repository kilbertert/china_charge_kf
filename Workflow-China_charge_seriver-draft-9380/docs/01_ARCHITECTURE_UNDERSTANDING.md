# 架构图核心理解 — 充电桩客服工作流

> 来源: `32-6.17（充电桩知识库）/架构框架.png`
> 用途: 新版 `China_charge_seriver.yml` 重构的输入基线。任何后续设计/施工必须与本文档定义的两轮 × 三类 × 三流矩阵对齐,如有偏离需先回到此文档讨论。

---

## 1. 总览:二维矩阵

整个客服系统被组织成 **"两轮对话 × 三类场景 × 三条后台流"** 的二维矩阵。

| 维度 | 切分 | 含义 |
|------|------|------|
| 时间 | 第一轮 / 第二轮 | 用户诉求从模糊收敛到具体 |
| 场景 | 路径A / 路径B / 路径C | 用户细化后的业务意图类型 |
| 后台 | 流程1 / 流程2 / 流程3 | 第一轮进来时并行跑的预检流 |

---

## 2. 时间维度:两轮对话

### 2.1 第一轮对话输入
- **入口**: 用户发起咨询,可能是文本 / 图片 / 语音
- **作用**: 不是为了"答问题",而是为了**把一句模糊的话先收敛成"用户大概想解决什么"**
- **后台动作**: 同步触发流程1 / 流程2 / 流程3 三个预检流(详见第 4 节)

### 2.2 第二轮对话
- **入口**: 第一轮梳理后,系统主动/被动地让用户细化诉求("你具体想了解哪一方面?")
- **作用**: 用户给出更明确的问题后,根据意图路由到三条业务路径之一(详见第 3 节)

### 2.3 后续轮次(架构图右侧注释)
- 架构图右侧黄色注释:**"如果客户开始询问其他,有以下个问题可通过流回输入到后台"**
- 含义: 第二轮之后如果用户又问新问题,允许**回流**到后台功能重新跑流程1/2/3,而不是强制两轮结束
- 施工含义: 多轮支持是 **MUST**,turn_id / session 状态必须可追踪

---

## 3. 场景维度:三条业务路径

第二轮时根据用户细化诉求,路由到以下三路之一。

### 3.1 路径 A — 功能查询 / 检查

| 项 | 内容 |
|----|------|
| 业务含义 | 售前咨询 + 功能性问答 + 兼容性 / 保修期类问题 |
| 典型问法 | "这桩有什么功能?" / "定时充电怎么开?" / "保修期多久?" / "我家电压能装吗?" |
| 处理方式 | KB 检索 → LLM 整理 → 输出功能清单 / 答案 |
| 关键区分 | 是**知识性问答**,不是动作性 step-by-step |
| 期望输出 | 文本回答 + 可选功能清单数组 |

### 3.2 路径 B — 业务 / 规则类问题诊断

| 项 | 内容 |
|----|------|
| 业务含义 | 售后故障 + 业务规则类(余额、退款、工单、合规) + 多步方案 |
| 典型问法 | "充电中断" / "余额没到账" / "申请退款怎么走" / "充电桩报故障码 E07" |
| 处理方式 | 故障/规则诊断 → 可能触发工单分流 → 输出多步方案 |
| 关键区分 | **必须先做危险信号硬判定**,再进入诊断;不能跳过 |
| 期望输出 | 诊断结论 + 处理步骤 + 可能的下一步入口(工单/人工) |

### 3.3 路径 C — 移动设备操作指导

| 项 | 内容 |
|----|------|
| 业务含义 | App / H5 / 小程序 / 扫码 / 远程控制的操作引导 |
| 典型问法 | "App 上怎么改语言" / "扫码后页面不动" / "怎么解绑车辆" |
| 处理方式 | 步骤化指引,按用户设备和语言输出 |
| 关键区分 | 是**动作性 step-by-step**,每一步都要可执行可验证 |
| 期望输出 | 编号步骤列表 + 可选截图说明 |

---

## 4. 后台维度:三条预检流(流程1/2/3)

第一轮对话进入后台时,**并行**跑以下三个查询流,结果作为第二轮的上下文。

### 4.1 流程 1 — 查询匹配功能是否已实现

- **作用**: 确认"用户问的功能是否真的在产品库/知识库内覆盖"
- **对应问题**: "这桩能不能定时充电?" → 系统先确认"定时充电"这个功能是否在该型号产品库中存在
- **失败处理**: 若未实现 → 明确告知用户"暂不支持"或"需升级到 XX 版本"
- **数据源**: 功能库(产品功能矩阵)

### 4.2 流程 2 — 查询功能清单是否准确无误

- **作用**: 对流程1已命中的功能,再校验"清单描述是否与最新版本匹配"
- **对应问题**: 即使流程1命中"定时充电",也要确认当前描述和最新固件/版本一致
- **失败处理**: 若有过时 → 触发知识库版本提醒(归档 + 重新标注)
- **数据源**: 清单校验库(版本/时效标签)
- **痛点**: 这是架构图黄色注释提到的"新增策略而检查麻烦"的核心位置——扩展时容易漏改

### 4.3 流程 3 — 查询报价

- **作用**: 单独查询产品价格、配件价格、保修期内/外报价
- **对应问题**: "这台 7kW 充电桩多少钱?" / "换一个充电枪头多少钱?"
- **独立原因**: 报价独立、敏感、可能动态(促销/区域),不能和功能流混在一起
- **数据源**: 报价库(独立 KB)

### 4.4 三流的关系

```
第一轮输入
    │
    ├──→ 流程1: 功能匹配 ──┐
    ├──→ 流程2: 清单校验 ──┼──→ 第二轮上下文(信息完备性)──→ 路径 A/B/C
    └──→ 流程3: 报价查询 ──┘
```

---

## 5. 底部黄色注释:两个已知痛点

### 5.1 "存在 新增策略而检查麻烦 / 常的差的问题"
- 含义: 知识库或规则扩展时,流程1/2/3 的检查逻辑会冗余/失效
- 施工含义: 检索策略要做成**可插拔**,新增一个功能/型号不能改流程1的判定逻辑

### 5.2 "跨文档与多渠道咨询问题"
- 含义: 同一问题可能跨多个文档(保修政策 + 产品手册 + 安装规范),同时用户可能从 H5 / App / 微信小程序 / 邮件多渠道进来
- 施工含义: 需要**统一意图入口**;多渠道共享同一个 scene 分类结果

---

## 6. 跨轮次/跨路径的"信息流"约束

为保证多轮一致性和可测试性,后台必须维护以下信息流:

1. **第一轮的三流结果**必须保留到第二轮(可放在 `conversation_variables` 或 `sys.query` 中传递)
2. **路径选择结果**必须可回放(`scene` 字段持久化)
3. **危险信号判定**一旦触发,必须立即分流,不能被后续路径覆盖
4. **多语言**不只是在 LLM prompt 里换 `{{language}}`,而是要影响路径选择(例如某些路径C只支持部分语言)

---

## 7. 验收检查项(用于后续 SPEC 拆解)

任何重构后的工作流 yml,必须能回答以下问题:

- [ ] 是否新增了 **scene_classifier** 节点(区分路径 A/B/C)?
- [ ] 是否新增了 **流程1/2/3** 三个独立的查询节点(而非混在一个 KR 里)?
- [ ] 是否支持 **多轮**(turn_id / session 状态)?
- [ ] **路径 B** 是否有独立的**危险信号硬判定 code 节点**(不能只靠 LLM prompt 文字描述)?
- [ ] **End 节点** 是否输出统一 SceneResponse JSON(便于前端按 scene 渲染)?
- [ ] **路径 C** 是否对每种目标语言独立处理?
- [ ] 新增功能/型号时,流程1的检索策略是否需要改 yml?
- [ ] 是否对 H5 / App / 多渠道共享同一 scene 分类?
