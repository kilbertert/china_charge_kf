## 四、会话变量状态机(核心)

### 会话变量清单(8 个)
| 变量 | 类型 | 现有/新增 | 作用 |
|---|---|---|---|
| cv_flow_state | string(枚举) | 🆕 | 二阶段主状态机 |
| cv_record_id | string | 🆕 | 当前相关行企微 record_id(N14/N19 用) |
| cv_row_summary | string | 🆕 | 旧行内容快照(N9 对比 + N17 锚点) |
| cv_feedback_zh | string | 🆕 | N5 转写标准化中文反馈(N9/N17 锚点) |
| cv_pending_clarify | bool | 现有 | L1 澄清回流 |
| cv_matched_module | string | 现有 | L1 板块 |
| cv_last_intent | string | 现有 | L4 a/b/c/d |
| cv_bug_logged | bool | 现有 | 兼容 |

### cv_flow_state 状态枚举
| 状态 | 触发节点 | 进入时动作 | 用户回应分支 | arm 定时器 |
|---|---|---|---|---|
| IDLE | 初始/完成 | — | 正常 L1→L4 路由 | cancel |
| AWAIT_CONFIRM_IDENTITY | N3输出旧行进度后 | 存 cv_record_id + cv_row_summary | 否认→N5 / 确认→AWAIT_DIFF | arm |
| AWAIT_DIFF | N9对比有差异(瞬时) | 存 cv_feedback_zh | 直接进 AWAIT_DIFF_DECISION | — |
| AWAIT_DIFF_DECISION | N11询问是否修改 | — | 确认修改→AWAIT_CONFIRM_MODIFY / 超时→N19 | arm |
| AWAIT_CONFIRM_MODIFY | N12完成修改汇报后 | — | 确认→N14写主表 / 不确认+改→回本态 / 超时→N19 | arm |
| AWAIT_CONFIRM_NEW | N6引导确认后(新增) | 存 cv_feedback_zh | 确认→N16写主表 / 继续反馈→回N6 / 超时→N19 | arm |

### 状态迁移图
```
IDLE ──N2命中──► AWAIT_CONFIRM_IDENTITY ──否认──► (回N5) ──► AWAIT_CONFIRM_NEW
 │                  │                                  ▲              │
 │                确认+补充                            │            确认
 │                  ▼                                  │              │
 │              AWAIT_DIFF ──有差异──► AWAIT_DIFF_DECISION           │
 │                  │                    │              │            │
 │            静默/无差异              确认修改           │          超时
 │                  │                    ▼              │            │
 │                END              AWAIT_CONFIRM_MODIFY │            │
 │                                  │      │     │      │            │
 │                             确认 │  超时  │不确认+改│            │
 │                                  ▼      ▼      ▼      │            │
 │                             N14写主表  N19  回AWAIT_CONFIRM_MODIFY │
 │                                  │                     │            │
 │                                  └────► IDLE ◄──────────┘            │
 │                                                                    │
 └──N2不存在──► N5转写 ──► AWAIT_CONFIRM_NEW ──确认──► N16写主表 ──► IDLE
                                  │
                             超时/继续反馈
                                  └──► N19缓存表(不写主表)
```

### 跨轮次续接机制(非同轮循环)
每轮 chatflow 入口(6001 Start 后)加状态分发 if-else,读 cv_flow_state:
- IDLE → 正常走 L1→L4 路由(现有逻辑)
- AWAIT_* → 先走 N17 相关性识别,判断本轮发言是"回应待确认事项"还是"开新话题":
  - 是回应 → 按 cv_flow_state 进对应确认分支(N4/N11/N13/N7)
  - 是新话题(IRRELEVANT) → reset 到 IDLE 正常路由,后端 cancel 旧 timer
  - 是 MODIFY_NEW → 进 N8(新增数据修改)
  - 是 MODIFY_EXISTING → 进 N9(旧行修改)

## 五、Timer arm/cancel 映射(后端侧)

后端 MessageProcessor 解析 Dify answer 末尾握手标记,按 cv_flow_state 对应 arm。

### 握手标记格式(answer 末尾,后端正则提取后剥离)
```
<!--SYS:TIMER|action=arm|state=await_confirm_new|record_id=RECxxx|ts=1782554840-->
<!--SYS:TIMER|action=cancel-->
```

### arm/cancel 规则
| cv_flow_state | Celery 任务 | countdown | fire 时动作 |
|---|---|---|---|
| AWAIT_CONFIRM_IDENTITY | bugtrack_timeout | 1800s | 写缓存表 N19,关联 cv_record_id |
| AWAIT_DIFF_DECISION | bugtrack_timeout | 1800s | 写缓存表 N19 |
| AWAIT_CONFIRM_MODIFY | bugtrack_timeout | 1800s | 写缓存表(旧行半成品) |
| AWAIT_CONFIRM_NEW | bugtrack_timeout | 1800s | 写缓存表(新增未确认,不写主表) |
| IDLE / 确认完成态 | — | — | cancel(revoke 旧 task_id) |

### PendingTimerStore(新建,类比 ConversationStore)
`{user_id} → {task_id, state, record_id, armed_at, payload}`,memory/redis 实现。
- **与 CLAUDE.md 无状态约束关系**:存的是待办定时器元数据(非会话历史),性质同已允许的 ConversationStore。边界需在 CLAUDE.md 注明。

### 定时器生命周期
```
用户发消息 → MessageProcessor → Dify chatflow 返回 answer
                │
      含 <!--SYS:TIMER|...--> 标记?
        ├─ arm → 剥离标记 + Celery.apply_async(countdown=1800, args=(payload,))
        │         → task_id 存 PendingTimerStore[user_id]
        │         → 发送 answer 给用户
        └─ cancel/终结 → 若该 user 有 pending timer → revoke(旧task_id)
                        → 发送 answer 给用户

[30分钟后无响应] Celery task fire → 读 payload → 按 state 写缓存表 N19

[30分钟内再说话] MessageProcessor 入站 → 先查 PendingTimerStore → 有 → revoke → 清除
              → 再正常走 Dify chatflow(靠 cv_ 状态走 N17 相关性分发)
```

## 六、N17 相关性识别 prompt 草案(用户后续自行填充)

当 cv_flow_state ∈ AWAIT_* 且用户新发言进来时触发。Dify 侧 LLM 节点,吃 cv_ 上下文。

```
你是一个客服对话相关性判定器。当前用户有一条【待确认事项】尚未完成,
用户在 30 分钟内又发了新消息。请判断这条新消息与待确认事项的关系。

【待确认事项类型】: {cv_flow_state}
  - await_confirm_identity: 我们刚告诉用户命中了一条已有 bug 记录,在问他"这是不是你反馈的问题"
  - await_diff_decision: 我们问用户"是否要修改这条已有记录"
  - await_confirm_modify: 我们刚把修改后的内容汇报给用户,在问他"确认无误吗"
  - await_confirm_new: 我们刚整理完用户的新反馈,在问他"确认这样记录吗"

【相关历史内容】(旧行快照): {cv_row_summary}
【相关历史内容】(用户已转写的反馈): {cv_feedback_zh}

【用户新消息】: {user_message}

请只输出一个分类标签,不要输出任何其他文字:
- IRRELEVANT       : 新消息与待确认事项无关(用户开了新话题/问了别的事)
- MODIFY_NEW       : 新消息是对【尚未写入的新反馈】的补充或修改要求
- MODIFY_EXISTING  : 新消息是对【已有 bug 记录】的修改要求(针对旧行内容)

判定要点:
1. 如果用户在回答"是不是这个问题/确认吗/改不改",属于回应待确认事项:
   - 这类回应由后续确认分支处理,不是本分类的目标,输出 IRRELEVANT 让状态机正常推进。
2. 如果用户提出新的问题内容、或要求增改反馈的某些字段,且针对的是"刚整理的新反馈" → MODIFY_NEW
3. 如果用户要求修改的是"命中那条已有记录"的内容 → MODIFY_EXISTING
4. 话题完全跳转(比如突然问计费规则) → IRRELEVANT
```
